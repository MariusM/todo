## Challenges

- **npm cache permissions (EACCES)** blocked vitest-axe installation in Story 4.4. Team pivoted to manual WCAG contrast tests and eventually installed axe-core directly. This environment issue could recur in Epic 5.
- **Code review still catching 3-5 substantive issues per story** — Space key scroll prevention (4.2), repeated announcement bug (4.3), O(n^2) algorithm (4.3), missing axe-core test (4.4). These are real defects, not style nits.
- **Pre-existing issues discovered in 3 of 4 stories** — vitest config, undefined color token, inadequate contrast ratios. Each epic stress-tests previous work, which is healthy but adds unplanned effort.

## Key Insights

1. **Audit-first approach saves effort** — checking existing behavior before coding prevented unnecessary work in Story 4.2 and should be applied to future stories where existing behavior may satisfy ACs
2. **Code review is the team's most valuable quality gate** — substantive bugs caught in every story, with decreasing severity over time (from missing features in Epic 2 to subtle aria-live edge cases in Epic 4)
3. **Each epic naturally stress-tests previous work** — pre-existing issues surfacing is healthy discovery, not failure. The codebase gets healthier with each epic.
4. **The two-commit pattern (implement → review fix) is accepted and effective** — consistent quality bar, every story ships clean after review
5. **Sequential accessibility stories build on each other** — the ordering (responsive → keyboard → SR → contrast) was ideal, each story leveraging foundations from the previous one
